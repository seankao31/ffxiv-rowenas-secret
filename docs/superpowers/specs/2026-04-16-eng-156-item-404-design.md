# Item Page 404 Design

## Summary

Navigating to `/item/<id>` with an item ID that doesn't exist in our dataset currently renders a broken page: "Item #9999999" header with empty market data, no icon, no recipe. Fix by having the load function reject unknown IDs with `error(404, 'Item not found')`, and add a root-level `+error.svelte` so the rejection renders a proper client-rendered error page instead of SvelteKit's default unstyled fallback.

## Approach

Three changes:

1. **Validate item ID in `src/routes/item/[id]/+page.server.ts`.** After `waitForNameCache()`, guard against an empty cache (5xx), then check whether the ID is in either the TW name map or the scanner's item cache. If in neither, throw `error(404, 'Item not found')`. The existing 400 check for non-integer IDs stays.

2. **Add `src/routes/+error.svelte`.** A root-level error boundary that renders status code and message. Catches errors from any route, not just item pages — pays off for unknown routes and future page errors. Displays a "Back to dashboard" link.

3. **Add `"5111": "鐵礦"` to `src/lib/server/fixtures/snapshot.json`'s `names` map** so the existing `tests/e2e/crafting-breakdown.test.ts` (which navigates to `/item/5111` for its non-craftable tab test) keeps working under the new 404 guard.

## Validity Oracle

An item is considered to "exist" if it's in **either**:

- The TW name map (`getNameMap()`), populated from `data/tw-items.msgpack`, **OR**
- The scanner's item cache (`getAllItems()`), populated from live scanning (production) or `snapshot.items` (fixture mode).

Rationale: the app's own dashboard can link to items without TW names. `src/lib/server/scoring.ts:166` emits `itemName: 'Item #<id>'` as a fallback when `nameMap.get()` is undefined, and `OpportunityTable.svelte:132` links every opportunity row to `/item/{id}`. During global→TW patch-lag windows, a freshly-scannable-but-untranslated item can generate such a row. Treating those IDs as non-existent would 404 links the app generates itself. The union predicate keeps those pages reachable while still 404-ing IDs that neither the name map nor the scanner knows about.

## Rendering Constraint: SPA Mode

The app sets `export const ssr = false` in `src/routes/+layout.ts`, making it a SPA. Under SvelteKit's documented behavior:

- The initial document response for `/item/<id>` is an empty shell with HTTP **200**, regardless of what the server load function does.
- The server load still runs; if it throws `error(404)`, the resulting error state is delivered to the client and `+error.svelte` renders client-side.
- Empirically verified: the existing `/item/abc` e2e test triggers `error(400)` and `response.status()` returns 200. The existing test pattern asserts on visible "400" text, not the HTTP status.

**Implication:** this ticket delivers a client-rendered 404 *experience*, not a document-level HTTP 404. That's consistent with the ticket's user complaint ("broken page" vs. "proper error page"). Real HTTP 404 semantics (for crawlers, link unfurlers, cache layers) require enabling SSR on this route — deferred to future work.

## Details

### Load function change

`src/routes/item/[id]/+page.server.ts`:

```ts
const parsed = Number(params.id)
if (!Number.isInteger(parsed) || parsed <= 0) {
  error(400, 'Invalid item ID')
}

await waitForNameCache()
const nameMap = getNameMap()
const itemCache = getAllItems()

if (nameMap.size === 0 && itemCache.size === 0) {
  error(503, 'Item data temporarily unavailable')
}

if (!nameMap.has(parsed) && !itemCache.has(parsed)) {
  error(404, 'Item not found')
}

const twName = nameMap.get(parsed) ?? null
const hasRecipe = getRecipesByResult(parsed).length > 0
return { itemID: parsed, twName, hasRecipe }
```

`twName` can still be `null` when the item exists only in `itemCache` (untranslated-but-scanned). The client's existing `resolveDisplayName()` fallback renders "Item #<id>" plus any English name from XIVAPI — current behavior preserved.

The empty-cache 503 guard is narrow: it only fires when *both* caches have been settled empty — a future fail-open pattern analogous to the existing `settleVendorPrices()`. It does **not** cover the app's current degraded-startup paths:

- If `fetchItemNames()` throws, `hooks.server.ts` calls `process.exit(1)` and no requests are served.
- If the scanner's `fetchMarketableItems()` retry loop can't make progress, `setNameMap()` is never called and requests block on `waitForNameCache()` indefinitely. Addressing this is an app-wide concern that belongs in a separate infrastructure ticket, not this per-route change.

The guard's value is intent-documenting: it ensures 404 means "unknown ID", not "data missing", if fail-open is ever introduced.

### Error page

`src/routes/+error.svelte` (new):

- Reads `$page.status` and `$page.error?.message`.
- Layout: large status code, message as subheading, "Back to dashboard" button linking to `/`.
- Styled with Tailwind + DaisyUI tokens to match existing pages. Uses `card` or similar to fit the app's visual language.
- Centered vertically/horizontally within the main content region.
- Responsive by default (no table or complex layout).

### Messages

- Unknown item: `error(404, 'Item not found')`
- Data unavailable: `error(503, 'Item data temporarily unavailable')`
- Malformed ID: `error(400, 'Invalid item ID')` (unchanged)

The error page displays whatever message is passed, so no per-error special-casing.

## Fixture Mode

`FIXTURE_DATA=true` (used by `bun run dev` for offline work and by Playwright — see `playwright.config.ts`) seeds both the name map and the item cache from `src/lib/server/fixtures/snapshot.json`. This is deliberately hermetic per ADR-014: fixture mode must work without network and without external data files.

Under the union predicate, items are reachable in fixture mode if they appear in either `snapshot.names` or `snapshot.items`. Synthetic test IDs `99901`/`99902` appear in both and remain reachable.

**Concrete fix:** add `"5111": "鐵礦"` (Iron Ore) to `snapshot.names`. `tests/e2e/crafting-breakdown.test.ts` navigates to `/item/5111` to verify the disabled Crafting tab on a non-craftable item. `5111` is absent from both `snapshot.names` and `snapshot.items`, so the test would regress without this entry. No corresponding `items` entry is needed — the test mocks market API responses.

Note on internal inconsistency: the test's XIVAPI mock labels `row_id: 5111, Name: 'Fire Crystal'` (line 23 of crafting-breakdown.test.ts), whereas real game 5111 is 鐵礦 / Iron Ore. This discrepancy predates ENG-156 and doesn't affect test assertions (which check tab state, not item names). Living with it; can be cleaned up separately.

**Policy going forward:** if a new e2e or dev-flow test needs an item not in the fixture, first check whether an existing fixture item can serve the test's intent. Only extend `snapshot.json` (either `names` or `items`) when no fixture item fits.

## Testing

### Unit tests (`tests/server/item-detail.test.ts`)

File already exists and tests the load function directly (imports `load` from the route module). Extend with:

- Known ID in name map → returns `{ itemID, twName, hasRecipe }` with `twName` set. (The existing non-craftable test uses id=9999 with name map `[[1, 'placeholder']]`; update it so 9999 is in the name map, or change the test id.)
- ID present in item cache but **not** in name map → load succeeds with `twName: null`. (Pins the untranslated-but-scanned path.)
- ID in neither name map nor item cache → throws `HttpError` with status 404.
- Both caches empty → throws `HttpError` with status 503.

The 400 cases for non-integer / negative / zero / decimal IDs already exist and stay unchanged.

Tests use `expect(...).rejects.toMatchObject({ status: 404 })` (or 503) following the pattern already in the file. Use `setItem(...)` to populate the item cache for the "untranslated-but-scanned" test (see `src/lib/server/cache.ts`).

### E2E tests (`tests/e2e/item-detail.test.ts`)

Add a scenario for an unknown ID, following the existing `/item/abc` test pattern (assert on visible text, not HTTP status):

- Navigate to `/item/9999999`.
- Assert "404" is visible.
- Assert the error message text is visible (e.g., "Item not found").
- Assert "Back to dashboard" link is present and points to `/`.
- Assert none of the normal item-page elements render (no Cross-World Listings card, no tab bar).

Mock API calls as needed to stay offline, following the existing pattern in `item-detail.test.ts`.

### Visual verification

Use the `playwright-cli` skill to navigate to `/item/9999999` and screenshot. Confirm the error page looks polished at both desktop and mobile viewports.

## Out of Scope

- Toast / flash-message infrastructure (no existing infra; premature to build for one use case).
- Item-specific error page (`src/routes/item/[id]/+error.svelte`). Root-level is sufficient; we can add per-route errors if future UX calls for them.
- Search suggestions, "did you mean…?", or recovery hints on the 404 page.
- Redirecting to `/` on unknown item — silently hides the bad URL from the user and breaks link-sharing.
- Wiring the name cache for fail-open (`settleNameCache()` analog) or adding a timeout fallback for `waitForNameCache()`. Today's degraded-startup path (scanner hangs retrying `fetchMarketableItems()`) makes every server load hang, not just the item route. Fixing it belongs in a separate infrastructure ticket that addresses the cache module holistically, not in this per-route change.
- Loading fixture names from `tw-items.msgpack`. Considered and rejected: breaks ADR-014 hermeticity (msgpack isn't checked in and requires network/build-step to obtain).
- Filtering untranslated items out of opportunity generation. Under the union predicate, those items remain reachable via `/item/[id]`, so no upstream filter is needed.

## Future Work

- **Enable SSR on `/item/[id]`** for true HTTP 404 semantics (crawlers, link unfurlers, monitoring). Requires verifying that the item page and its child components (ListingsTable, SaleHistoryTable, PriceStats, CraftingBreakdown) are SSR-safe. Separate ticket when the need for HTTP-level 404s materializes.
- **Name cache failure path** — a bounded-timeout or explicit-settle mechanism for `waitForNameCache()` so degraded startup returns 5xx instead of hanging all server loads. Infrastructure work, not route-specific.
