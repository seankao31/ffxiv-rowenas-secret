# Fixture Data Dev Mode

**Ticket:** ENG-118
**Status:** Approved

## Problem

Every dev server start requires a full Universalis scan (~7,000 items across 8 worlds) before the UI renders any data. For UI-only iteration, this is unnecessary overhead.

## Solution

Add a `FIXTURE_DATA=true` environment variable that pre-seeds the in-memory cache with a snapshot of real market data and skips the scanner entirely. The `/api/opportunities` endpoint returns scored results immediately on first request.

## Trigger

- Environment variable: `FIXTURE_DATA=true`
- Checked in `src/hooks.server.ts` `init()` before `startScanner()` is called
- When set: call `seedFixtureData()` instead of `startScanner()`
- Recipe loading (`initRecipes()`) and vendor prices (`fetchVendorPrices()`) still run normally

## Fixture Data Source

A JSON snapshot file at `src/lib/server/fixtures/snapshot.json` containing:

- 30-50 `ItemData` objects with realistic listings, history, and upload times
- Corresponding name mappings (itemID -> name)

Generated by a one-time helper script (`scripts/snapshot-cache.ts`) that captures a slice of a real scan. The snapshot is checked into the repo so any developer can use fixture mode without running the scanner first.

## Cache Seeding

New module: `src/lib/server/fixtures/seed.ts`

`seedFixtureData()` function that:

1. Reads and parses `snapshot.json`
2. Calls `setItem()` for each item in the snapshot
3. Populates the name cache via existing cache API
4. Sets `scanMeta` to indicate the cache is ready (so `/api/opportunities` returns `{ready: true}` instead of HTTP 202)

This function is synchronous from the caller's perspective — the cache is populated before `init()` returns.

## Startup Flow

```
hooks.server.ts init():
  initRecipes()                    // unchanged
  fetchVendorPrices()              // unchanged (background)

  if (process.env.FIXTURE_DATA) {
    seedFixtureData()              // new: instant cache population
  } else {
    startScanner()                 // existing: async scan loop
  }
```

## Snapshot Generation Script

`scripts/snapshot-cache.ts`:

- Imports scanner/Universalis fetching logic to run a single scan pass
- Selects a representative subset (~30-50 items) from the results
- Serializes to JSON and writes to `src/lib/server/fixtures/snapshot.json`
- Run manually: `bun run scripts/snapshot-cache.ts`

This script is run ad-hoc by developers when the fixture data needs refreshing. It is not part of the build or dev pipeline.

## What Doesn't Change

- Scanner code (`src/lib/server/scanner.ts`) — untouched
- `/api/opportunities` endpoint — untouched, reads from cache as usual
- Cache module (`src/lib/server/cache.ts`) — no structural changes, may need to export setters if not already available
- Production behavior — `FIXTURE_DATA` is never set in production environments
